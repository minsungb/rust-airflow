# 문서 작성 및 활용 가이드

프로젝트 전반의 문서 일관성을 유지하기 위해 README, 예제 시나리오, 세부 가이드 작성 시 다음 원칙을 따릅니다.

## 기본 원칙

1. **최신성 유지**: 코드/기능 변경 시 관련 문서의 예제 경로, 명령어, 화면 설명을 즉시 갱신합니다.
2. **한국어 우선**: 사용자 매뉴얼은 한국어로 작성하되, API 이름 등 고유명사는 원문을 그대로 표기합니다.
3. **파일 위치 명시**: 독자가 바로 파일을 찾을 수 있도록 상대 경로(`src/engine.rs` 등)를 본문에 기재합니다.
4. **실행 가능 예시 제공**: `cargo run --release`와 같이 실제 동작이 확인된 명령어만 포함합니다.

## README.md 업데이트 가이드

- 새로운 기능이나 구성 요소를 추가했다면 `## 주요 기능` 혹은 `## 프로젝트 구조` 섹션을 확장합니다.
- 빌드/실행 방법이 늘어났다면 `## 실행 방법`에 하위 섹션을 만들어 플랫폼별 명령어를 구분합니다.
- 문서나 시나리오 샘플 파일을 추가한 경우 `## 문서 사용 안내`에 링크를 추가해 독자가 쉽게 탐색하도록 합니다.

## docs/ 디렉터리 구성

| 파일 | 목적 |
| --- | --- |
| `docs/README.md` | 바이너리 자산 안내. 필요 시 스크린샷 위치 등 메모를 추가합니다. |
| `docs/DOCUMENTATION_GUIDE.md` | 문서 작성 규칙(본 문서). |
| `docs/<기능>-guide.md` | 기능별 심화 가이드. 예: `docs/sqlldr-guide.md`. |

새로운 Markdown 파일을 추가할 때는 표 상단에 행을 추가하고 목적을 간략히 적어둡니다.

## 예시 목차

아래 목차는 기능 가이드 문서에 공통으로 적용할 수 있는 템플릿입니다.

1. 소개
2. 필요 조건
3. 설정 방법
4. 실행 절차
5. 문제 해결(FAQ)
6. 관련 파일/경로 링크

필요에 따라 섹션을 추가하되, 위 순서를 기본으로 유지하면 독자가 탐색하기 쉽습니다.

## 기여 워크플로

1. 변경할 문서를 편집합니다.
2. `cargo fmt` 또는 관련 포맷터로 코드 블록을 정리합니다.
3. `git status`로 변경 사항을 확인하고, 커밋 메시지에 어떤 문서를 업데이트했는지 명시합니다.
4. PR 설명에 문서 변경 목적과 검증 내용을 요약합니다.

문서만 변경하더라도 리뷰어가 참고할 수 있도록 변경 이유와 테스트 범위를 간단히 기술해 주세요.
